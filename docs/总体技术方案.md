# 医疗情报系统总体技术方案

## 1. 文档目标

本文档基于现有需求文档与 `总体设计V1.md` sketch，给出首版原型系统的总体技术方案。目标是为后续实现提供清晰的系统边界、模块划分、关键接口、数据流转方式和工程化约束。

当前方案默认面向 V1 原型，优先保证核心链路跑通、结构可扩展、证据可追溯，不在首版引入不必要的系统复杂度。

## 2. 对现有 V1 Sketch 的判断

### 2.1 已有方向是正确的

`总体设计V1.md` 已经抓住了几个关键点：

- 以多数据源连接器为基础能力
- 在业务层引入统一 schema
- 使用多阶段生成报告，而不是一次性生成
- 意识到不同阶段应采用不同检索策略

这些判断说明系统主线是合理的，尤其是“连接器 + 统一数据模型 + 多阶段报告生成”这一思路，应当保留。

### 2.2 当前 sketch 的主要缺口

但现有 sketch 还不足以指导实现，主要缺少以下内容：

- 缺少 `Web/API` 交互入口与请求编排，尚未覆盖需求中的交互方式
- 缺少缓存与存储层，统一 schema 没有实际落地载体
- 缺少原始数据、规范化数据、最终报告之间的关系设计，证据追溯路径不完整
- “相关度筛选”放在采集侧过早，存在误删关键证据的风险
- LLM 层直接绑定 `OpenRouter`，扩展性和替换成本较高
- 缺少配置管理、日志、错误处理、上下文窗口治理、测试策略等工程化设计

因此，建议保留你 V1 sketch 的主干思想，但需要补齐“系统边界、存储、编排、可追溯、工程化”这几个关键层。

## 3. V1 总体定位

### 3.1 目标

V1 以“可工作的科研情报原型系统”为目标，支持用户输入靶点名称后，自动完成多源数据采集、结构化整理、LLM 分析和报告生成。

### 3.2 V1 范围

V1 聚焦核心链路：

- 用户输入靶点
- 系统调用 `ClinicalTrials.gov` 与 `PubMed`
- 对采集结果做规范化、去重、排序和统计
- 使用 LLM 分阶段生成结构化情报报告
- 在 Web 页面展示报告，并支持导出 Markdown

V1 不纳入以下能力：

- 独立异步任务队列或 worker
- 历史版本对比
- 图表可视化
- PDF 导出
- 多个高级外部数据源的批量接入

这些能力在架构上预留接口，但不在首版实现。

## 4. 技术选型

### 4.1 总体形态

建议采用“模块化单体”架构：

- 前端：轻量 Web 页面
- 后端：单个 API 服务
- 存储：本地规范化缓存数据库
- LLM：通过 Provider 抽象层接入外部模型服务

这样做的原因是：

- 首版原型开发效率高
- 容易满足需求中的交互入口要求
- 后续新增数据源时无需拆分系统
- 避免过早引入微服务、消息队列、任务系统等复杂度

### 4.2 推荐栈

- 后端：Python 3.12 + FastAPI
- 数据采集：`httpx`
- 数据建模：`Pydantic`
- 存储：`SQLite` + `SQLAlchemy`
- 前端：React + Vite
- 报告渲染：Markdown 转 HTML
- 配置管理：`.env` + `pydantic-settings`
- 日志：Python 标准 logging，输出结构化字段

### 4.3 LLM 接入策略

不建议在业务逻辑中直接写死 `OpenRouter`。建议抽象一个统一 Provider 接口，支持兼容 OpenAI 风格的调用协议。

建议设计：

- `LLMClient`：统一的模型调用接口
- `OpenRouterProvider`：首个可用实现
- 后续可增加 `OpenAIProvider` 或其他 provider

这样可以在不修改业务逻辑的情况下切换模型服务。

## 5. 系统架构

### 5.1 模块划分

建议后端按以下分层组织：

- `api`
  - 对外 REST 接口
  - 请求校验与响应封装
- `orchestration`
  - 核心流程编排
  - 串联检索、规范化、分析、报告生成
- `connectors`
  - 各外部数据源连接器
  - 每个数据源单独实现
- `normalize`
  - 原始数据到中间标准化层的映射
  - 去重、候选字段抽取、枚举归一
- `analyze`
  - 统计计算、相关性排序、证据裁剪
- `report`
  - 分阶段摘要生成
  - 最终报告拼装与 Markdown 输出
- `infra`
  - 配置、日志、数据库、异常处理、重试策略

### 5.2 前端职责

前端只承担轻量交互，不承载复杂业务逻辑，主要负责：

- 输入靶点名并提交查询
- 展示生成中的状态
- 展示最终报告
- 展示证据来源列表
- 支持复制或下载 Markdown

## 6. 核心数据分层与 schema 策略

### 6.1 查询对象

`TargetQuery`

- `target`: 用户输入的靶点名
- `aliases`: 解析出的常见别名或同义词
- `indication`: 可选适应症上下文
- `generated_at`: 查询发起时间

### 6.2 原始记录

`RawRecord`

- `source_name`
- `source_id`
- `payload`
- `retrieved_at`
- `query_snapshot`

原始记录的作用不是直接喂给 LLM，而是保留真实采集证据，支撑回溯与调试。

### 6.3 规范化记录（Schema TBD）

当前阶段，不建议冻结 `TrialRecord` 与 `LiteratureRecord` 的最终字段表。

原因是：

- 这些字段最终是否有价值，应由四类任务反推，而不是直接从源接口字段平铺得出
- `ClinicalTrials.gov` 与 `PubMed` 擅长的信息维度不同，强行提前对齐容易损失信息
- 还需要先完成“任务-字段候选矩阵”与小规模样本验证，才能知道哪些字段应进入稳定标准化层

因此，现阶段只冻结两类概念对象，具体字段均记为 `TBD`：

- `NormalizedTrialRecord`: `TBD`
- `NormalizedLiteratureRecord`: `TBD`

但它们需要满足以下约束：

- 必须保留来源追溯信息，如 `source_name`、`source_id`、`source_url`
- 必须区分“源字段原样映射”和“跨源归一后的语义字段”
- 必须允许不同来源在字段覆盖度上不完全一致
- 必须支持后续面向任务的派生特征补充，而不是把标准化层等同于最终分析层

### 6.4 聚合对象（Schema TBD）

当前阶段，也不建议冻结 `EvidenceBundle` 的最终字段表。

聚合对象应承载的是“面向任务的分析输入”，而不是简单把试验记录和文献记录拼接到一起。

因此，聚合对象现阶段定义为概念层，具体字段记为 `TBD`：

- `AnalysisReadyBundle`: `TBD`

但其职责可以先冻结：

- 绑定当前 `TargetQuery` 上下文
- 汇总已选中的试验与文献证据引用
- 保存由程序计算出的统计结果
- 保存 coverage、缺失项、warnings 等可解释性信息
- 作为报告生成模块的唯一上游输入，避免后续模块直接依赖某个特定数据源格式

### 6.5 最终产物

`ReportDocument`

- `report_id`
- `target`
- `summary_sections`
- `markdown_content`
- `source_refs`
- `generated_at`
- `warning_summary`

## 7. 数据流转过程

### 7.1 靶点解析

系统收到靶点后，先进入 `TargetResolver`：

- 保留用户原始输入
- 扩展常见别名和同义词
- 生成统一查询上下文

这样做的目的是避免因命名差异导致漏检，例如只输入 `HER2` 时遗漏 `ERBB2`。

### 7.2 多源检索

连接器分别执行 source-specific 查询：

- `ClinicalTrialsGovConnector`
- `PubMedConnector`

每个连接器独立负责：

- 查询参数构建
- 分页处理
- 原始响应读取
- 异常处理
- 原始记录封装

多个数据源之间不得互相耦合，新增数据源时只增加新的 connector。

### 7.3 规范化与去重

`Normalizer` 负责把各源数据转为中间标准化表示，并完成：

- 来源追溯信息保留
- 任务相关候选字段抽取
- 日期格式统一
- 研发阶段枚举归一
- 机构/公司名称规范化
- 跨源关联线索提取，例如 DOI、NCT 号、databank accession
- 去重与重复记录合并

其中，精确字段集合以后续“任务-字段候选矩阵”验证结果为准，当前保持 `TBD`。

### 7.4 分析与裁剪

V1 不建议在采集阶段做激进过滤，而是在规范化后做“排序和裁剪”：

- 根据靶点匹配度排序
- 根据发布时间、研究阶段、文献新近性等排序
- 为 LLM 选取最有代表性的证据集
- 用代码直接计算统计指标

这样既能控制上下文长度，也能降低误删重要信息的风险。

### 7.5 报告生成

报告生成建议拆成四个阶段：

1. 生成靶点概述与机制简介
2. 生成在研管线概览与阶段分布
3. 生成近期研究动态摘要
4. 生成竞争格局判断、风险与机会

每个阶段都使用结构化中间结果，而不是直接从海量原始文本生成最终报告。

### 7.6 输出与展示

最终输出以 Markdown 为标准格式：

- 便于存储与版本化
- 便于渲染为 HTML
- 后续也可转换为 PDF

前端负责将 Markdown 渲染为可读页面，并展示对应来源链接。

## 8. API 设计

V1 建议提供以下接口：

### 8.1 创建报告

`POST /api/reports`

请求体：

- `target`
- `indication` 可选
- `force_refresh` 可选

返回：

- `report_id`
- `target`
- `markdown_content`
- `warnings`

### 8.2 查询报告

`GET /api/reports/{id}`

返回结构化报告内容与摘要统计。

### 8.3 查询来源证据

`GET /api/reports/{id}/sources`

返回与该报告关联的试验和文献记录，便于追溯来源。

## 9. 大数据量与上下文窗口处理

需求文档明确要求考虑“检索结果数据量很大时如何处理”。建议采用以下策略：

- 不把全量原始数据直接发送给 LLM
- 先通过规则和轻量打分筛出高价值记录
- 先按来源做中间摘要，再做全局综合
- 统计类结果尽量由程序直接计算
- LLM 只负责解释、归纳和竞争态势判断

这一策略本质上是分层摘要或 map-reduce，可显著降低上下文窗口压力，并减少幻觉风险。

## 10. 工程化要求落地

### 10.1 错误处理

对以下场景需要显式处理：

- 网络超时
- 外部 API 不可用
- 返回为空
- 单一数据源失败
- 字段缺失或结构变化

策略上应允许“部分失败但整体可用”，即：

- 某个 source 失败时，系统仍生成报告
- 在报告和日志中写明 warnings

### 10.2 配置管理

- API Key 不写入源码
- 所有敏感配置由环境变量注入
- 使用统一 settings 对象读取配置

### 10.3 日志

日志至少记录：

- 请求 ID
- 靶点名称
- 调用数据源
- 接口耗时
- 拉取记录数
- 最终纳入分析的记录数
- warning 数量

## 11. 测试方案

V1 至少覆盖以下测试层次：

- 连接器单元测试
  - 查询参数构造
  - 响应字段解析
  - 空结果与异常处理
- 规范化单元测试
  - 日期格式化
  - 阶段映射
  - 去重逻辑
- 报告生成集成测试
  - 使用固定 fixtures 生成 `HER2` 报告
  - 校验四个必备章节是否存在
  - 校验来源引用是否可追溯
- 人工验收
  - `HER2`
  - `KRAS G12C`

## 12. 主要技术风险与应对

### 12.1 靶点别名导致漏检

风险：

- 不同数据库使用不同命名方式

应对：

- 引入 `TargetResolver`
- 保留原名并扩展别名检索

### 12.2 数据源字段差异大

风险：

- 不同来源结构不一致，难以统一分析

应对：

- 明确 `RawRecord -> source_normalized -> analysis_ready` 三层模型
- 标准化层字段先保持 `TBD`，由任务驱动逐步冻结
- 所有下游模块只消费中间标准化层或分析层，不直接耦合源字段

### 12.3 LLM 幻觉或总结失真

风险：

- 生成内容可能脱离原始证据

应对：

- 统计数据由程序计算
- 报告中保留来源引用
- 使用分阶段生成而不是一次性大总结

### 12.4 上下文窗口超限

风险：

- 文献与试验结果较多时，无法直接整体输入模型

应对：

- 排序裁剪
- 分源摘要
- 分阶段合成

## 13. V2 演进方向

在 V1 跑通后，可按以下顺序扩展：

1. 接入异步任务与状态查询
2. 增加 FDA/EMA/会议摘要等数据源连接器
3. 引入历史报告版本管理
4. 增加图表可视化
5. 增加 PDF 导出

## 14. 结论

建议以“模块化单体 + 多连接器 + 任务驱动的中间标准化层 + 分阶段报告生成”作为 V1 的正式实现路线。

你当前的 `总体设计V1.md` 适合作为初步 sketch，但还不够支撑实现。更完整的落地版本应以本文档为准：在保留你原有主干思路的基础上，补齐 Web/API、缓存存储、证据追溯、Provider 抽象和工程化横切能力。
