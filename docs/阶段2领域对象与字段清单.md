## 1. 文档目的

本文档用于冻结“阶段 2：标准化、去重与分析输入构建”的对象边界，并给出一份可直接指导实现的字段清单。

当前文档解决的问题是：

- 哪些领域对象需要先进入代码库
- 每个对象负责哪一层数据

## 2. 本轮冻结的设计决策

### 2.1 `AnalysisReadyBundle` 从阶段 2 开始就内置四章节输入

`AnalysisReadyBundle` 不是一个简单的“试验列表 + 文献列表”容器，而是后续报告生成的唯一上游输入。

因此从阶段 2 开始，直接内置四个强类型章节输入：

- `target_overview`
- `pipeline_overview`
- `research_update`
- `competition_assessment`

### 2.2 章节输入只保存 key 引用，不保存正文 excerpt

当前版本的 `section_inputs` 仅保存：

- 进入该章节的 `trial_keys`
- 进入该章节的 `literature_keys`
- 程序计算得到的强类型 `facts`
- 选择与裁剪说明

当前不在 `section_inputs` 内重复保存正文摘录，原因是：

- 避免 bundle 内出现第二份半结构化文本副本
- 避免在 excerpt 表达形式尚未定稿前过早冻结 schema
- 让阶段 2 先稳定在“结构化事实 + 证据引用”这一层

### 2.3 `NormalizedLiteratureRecord` 暂不收窄作者、机构、资助字段

文献侧保留：

- 作者
- 机构
- 资助
- databank

这些字段虽然不一定全部在当前报告模板中直接使用，但后续分析阶段可能会用到，因此本轮先保留。

### 2.4 阶段 2 不把 `CrossSourceLink` 作为正式领域对象

当前阶段保留轻量关联线索，但不单独建立正式对象。

因此：

- 文献上的 `linked_nct_ids`、`databanks` 继续保留
- 分析层可以临时利用这些线索做统计
- 暂不引入 `CrossSourceLink` 持久化模型和独立对象生命周期

## 6. 当前明确不纳入的对象

本轮不建立正式对象：

- `CrossSourceLink`

原因：

- 当前阶段只需要轻量关联线索，不需要完整对象生命周期
- 深度文献-试验关联暂不作为阶段 2 主目标
- 等后续确实出现稳定业务场景，再决定是否抽成正式实体

## 7. 实现顺序建议

建议按以下顺序落地：

1. 先定义值对象与领域对象
2. 再实现 `trial_normalizer.py` 与 `literature_normalizer.py`
3. 再实现 `stats.py`、`selector.py`、`bundle_builder.py`
4. 最后再决定哪些对象需要单独做持久化模型

这样做的好处是：

- 先冻结业务协议，再冻结存储协议
- 避免字段仍在调整时把数据库表过早锁死
- 让阶段 2 验证重点集中在“对象边界是否支撑四章节报告”

## 8. 本轮持久化决策

在完成阶段 2 的对象定义、normalizer 和 bundle builder 之后，本轮实际选择落地以下持久化模型：

| 表名 | 对应对象 | 当前保存策略 | 说明 |
| --- | --- | --- | --- |
| `normalized_trial_records` | `NormalizedTrialRecord` | 每次按 `fetch_run_id` 重建替换 | 保存完整 JSON 载荷，同时保留 `trial_key`、`nct_id` 便于检索 |
| `normalized_literature_records` | `NormalizedLiteratureRecord` | 每次按 `fetch_run_id` 重建替换 | 保存完整 JSON 载荷，同时保留 `literature_key`、`pmid`、`doi` 便于检索 |
| `analysis_snapshots` | `AnalysisReadyBundle` | 每个 `fetch_run_id` 保留一个最新快照 | 保存完整 bundle JSON，作为后续报告生成的唯一上游输入 |

本轮没有单独建表的对象：

- `SectionInputBundle`
- 四类章节 `facts`
- `CoverageSnapshot`
- `WarningItem`
- `SourceTrace`
- `NormalizedDate`
- 其他值对象

原因是：

- 这些对象当前更适合作为 bundle 或标准化记录中的组成部分
- 现阶段把它们拆成独立表会明显增加 schema 复杂度
- 在字段仍可能细调的阶段，优先保存完整 JSON 更稳妥

当前持久化策略可以概括为：

- 试验标准化记录单独存
- 文献标准化记录单独存
- 章节输入、统计结果、coverage 与 warning 收敛到 `analysis_snapshots`

等阶段 3、阶段 4 明确出现以下需求时，再考虑进一步拆表：

- 需要对 sponsor、phase、journal 等字段做高频复杂查询
- 需要跨 `fetch_run` 复用和比较标准化记录
- 需要对章节 facts 做单独缓存或版本管理

## 9. LLM 增强对象、评分与持久化

本节用于冻结阶段 2 的 LLM 增强层设计。当前约束是：

- `NormalizedTrialRecord` 与 `NormalizedLiteratureRecord` 继续保持确定性标准化层语义
- LLM 结果使用独立增强对象保存，不回写标准化对象
- LLM 增强默认内嵌在 `AnalysisPipelineService.build()` 中自动执行
- 单条记录 LLM 失败时按“最佳努力继续”处理，不阻断整体分析快照生成

### 9.3 评分冻结规则

阶段 2 的评分统一分为三层：

- `rule_scores`
- `llm_scores`
- `final_scores`

统一采用四维度加总分：

- `target_overview`
- `pipeline_overview`
- `research_update`
- `competition_assessment`
- `overall_score`

融合规则冻结为：

- `final_dimension_score = 0.65 * rule_dimension_score + 0.35 * llm_dimension_score`

`Trial` 总分权重固定为：

- `target_overview = 0.15`
- `pipeline_overview = 0.35`
- `research_update = 0.10`
- `competition_assessment = 0.40`

`Literature` 总分权重固定为：

- `target_overview = 0.30`
- `pipeline_overview = 0.10`
- `research_update = 0.40`
- `competition_assessment = 0.20`

规则分要求：

- 明确按四维度分别产出
- 统一归一到 `0-100`
- `selector` 默认优先按 `final_scores.<dimension>` 排序
- 若某条记录没有成功生成 LLM 增强，则回退到对应的 `rule_scores.<dimension>`

### 9.4 调用与失败策略

阶段 2 的 LLM 增强调用方式冻结为：

- 调用入口：`AnalysisPipelineService.build()`
- 调用方式：逐条记录、同步调用
- 请求输入：仅使用标准化记录字段与 `TargetQuery`
- 请求接口：统一走 `build_llm_client()` + `LLMClient.generate_structured()`
- 返回格式：严格结构化 JSON Schema，不做自由文本后解析

失败策略冻结为：

- 单条记录失败时，仅为该记录写入 warning
- 成功记录照常进入 enrichment 列表和持久化表
- 失败记录不写 enrichment 表
- `AnalysisReadyBundle` 仍照常生成
- `llm_enrichment_summary` 需要给出总数、成功数、warning 数和使用的 prompt version
