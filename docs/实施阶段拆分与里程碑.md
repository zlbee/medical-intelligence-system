# 医疗情报系统实施阶段拆分与里程碑

## 1. 文档目标

本文档基于 [架构设计方案](D:\Project\REFERENCE\medical-intelligence-system\docs\架构设计方案.md)，给出一份能够直接指导落地实施的阶段拆分方案，用于回答以下问题：

- 系统按什么顺序实施最稳妥
- 每个阶段具体做什么
- 每个阶段的交付物是什么
- 何时算达到一个里程碑
- 各阶段之间的依赖关系与并行方式是什么

本文档默认面向 V1 原型系统，目标是尽快交付一个“可演示、可验证、可继续扩展”的版本。

## 2. 规划假设

为使阶段拆分足够具体，本文档采用以下默认假设：

- 团队规模：`2-4` 人小团队
- 研发方式：按周推进，短迭代交付
- 系统范围：以 `ClinicalTrials.gov + PubMed + LLM 报告 + Web/API + Docker` 为 V1 范围
- 部署方式：优先使用 `Docker Compose`
- 存储方式：V1 使用 `SQLite`
- 交付要求：除源代码外，还需要架构文档、运行说明和至少一个示例报告

如果后续团队规模或时间窗口变化，可以在不改主顺序的前提下调整每个阶段的持续时间。

## 3. 实施原则

### 3.1 先打通主链路，再补强体验

优先保证以下主链路跑通：

- 输入靶点
- 双源采集
- 标准化与去重
- 统计与裁剪
- LLM 生成报告
- Web/API 展示结果

图表、异步任务、历史版本等增强能力放在 V1 之后处理。

### 3.2 每个阶段都必须可验收

每个阶段结束时，都要有可视、可测或可演示的结果，而不是只完成“内部代码搭建”。

### 3.3 采用纵向切片推进

尽量避免先把所有层都写空壳，再统一联调。推荐采用纵向切片方式：

- 先完成最小后端骨架
- 再打通单条完整数据链路
- 再补全第二个数据源与完整报告链路
- 最后做前端与工程化封装

### 3.4 里程碑以“可交付物”为准

里程碑判断标准不以“代码量”或“模块数量”为准，而以是否形成稳定交付物为准，例如：

- 可以启动的容器环境
- 可以成功拉取的真实数据
- 可以生成的结构化报告
- 可以通过浏览器查看的演示页面

## 4. 总体阶段与里程碑总览

建议将 V1 实施拆为 `6` 个阶段、`6` 个里程碑。


| 阶段   | 建议时长     | 目标              | 主要输出                        | 里程碑            |
| ---- | -------- | --------------- | --------------------------- | -------------- |
| 阶段 0 | 3-4 个工作日 | 完成项目底座与开发环境初始化  | 项目骨架、Docker 骨架、配置与日志基础设施    | `M0` 项目可启动     |
| 阶段 1 | 5-7 个工作日 | 打通多源采集与原始数据入库   | 两个 Connector、原始记录存储、抓取验证    | `M1` 双源采集可用    |
| 阶段 2 | 5-7 个工作日 | 完成标准化、去重、分析输入构建 | 统一领域模型、标准化逻辑、统计与裁剪          | `M2` 分析数据可用    |
| 阶段 3 | 5-7 个工作日 | 打通报告生成与后端 API   | LLM Provider、四章节生成、REST API | `M3` 后端闭环跑通    |
| 阶段 4 | 4-5 个工作日 | 完成前端展示与容器化联调    | Web 页面、报告展示、Compose 联调      | `M4` Web 原型可演示 |
| 阶段 5 | 4-6 个工作日 | 完成稳定化、测试、示例交付   | 自动化测试、示例报告、运行文档             | `M5` V1 可提交验收  |


如按上述节奏推进，V1 一般可在 `5-7 周` 内形成较完整的原型交付。

## 5. 分阶段实施方案

## 5.1 阶段 0：项目底座与开发环境初始化

### 阶段目标

建立可持续开发的项目基础，确保团队在统一环境中协作，并为后续模块落地预留目录与基础设施。

### 范围

- 初始化后端目录与前端目录
- 搭建基础 Docker 环境
- 建立配置、日志、数据库初始化机制
- 确定基础代码规范与运行方式

### 具体任务

#### 后端

- 创建 `backend/app` 基础目录结构
- 初始化 `FastAPI` 入口文件
- 建立 `settings` 配置对象
- 建立 `logging` 初始化逻辑
- 建立 SQLite 连接与建库脚手架

#### 前端

- 初始化 `React + Vite` 项目
- 建立基础页面路由框架
- 准备 API 调用封装目录

#### 容器与环境

- 编写 `backend/Dockerfile`
- 编写 `frontend/Dockerfile`
- 编写 `docker-compose.yml`
- 准备 `.env.example`
- 确定数据库、缓存、报告输出挂载目录

#### 工程化

- 初始化 `pytest`
- 准备基础 `README` 或运行说明骨架
- 建立统一异常类与错误响应格式约定

### 交付物

- 可运行的前后端项目骨架
- 可启动的 `docker-compose.yml`
- 基础配置文件与环境变量模板
- 后端健康检查接口，如 `/health`

### 里程碑 `M0`

团队成员可通过统一命令启动基础环境，并访问：

- 后端健康检查接口
- 前端初始页面

### 验收标准

- `docker compose up` 后前后端容器均可启动
- 后端可连接到 SQLite 文件
- 配置读取不依赖硬编码密钥
- 日志可输出基础请求信息

## 5.2 阶段 1：多源采集与原始数据入库

### 阶段目标

打通“靶点输入 -> 双源真实数据拉取 -> 原始记录持久化”的采集链路。

### 范围

- 实现 `ClinicalTrials.gov` Connector
- 实现 `PubMed` Connector
- 定义 `RawRecord` 模型与原始数据存储
- 完成至少一个靶点的真实抓取验证

### 具体任务

#### `connectors`

- 定义 `BaseConnector` 协议
- 实现 `ClinicalTrialsGovConnector`
- 实现 `PubMedConnector`
- 处理查询参数构建、分页、超时、重试和异常返回

#### `domain`

- 定义 `TargetQuery`
- 定义 `RawRecord`
- 定义 `ConnectorResult`

#### `repository`

- 建立原始记录表结构
- 保存来源名、来源主键、来源 URL、原始载荷、查询快照

#### `orchestration`

- 实现最小 `FetchPipeline`
- 支持按靶点触发双源抓取

#### 测试

- 为两个 Connector 准备固定响应 fixture
- 覆盖空结果、接口异常、字段缺失场景

### 交付物

- 两个可调用的 Connector
- 原始记录持久化能力
- 以 `HER2` 或 `KRAS G12C` 为例的抓取结果样本

### 里程碑 `M1`

系统能够对给定靶点，从两个公开来源抓到真实数据，并将原始结果落入本地数据库。

### 验收标准

- 输入一个靶点后可返回双源抓取记录数
- 单一数据源失败时，不影响另一数据源的抓取结果保存
- 原始数据具备可回溯字段
- Connector 层有基本单元测试

## 5.3 阶段 2：标准化、去重与分析输入构建

### 阶段目标

把“原始数据”转成“下游可消费的数据”，形成标准化层和分析层。

### 范围

- 建立标准化领域对象
- 实现基础字段映射与归一
- 实现去重、跨源关联与排序裁剪
- 构建 `AnalysisReadyBundle`

### 具体任务

#### `normalize`

- 实现 `trial_normalizer.py`
- 实现 `literature_normalizer.py`
- 实现日期、阶段、状态、机构名称等归一逻辑
- 提取 `pmid`、`doi`、`nct_id` 等关联字段

#### `analyze`

- 实现靶点匹配评分
- 实现试验阶段分布统计
- 实现主要申办方统计
- 实现文献时间排序与高价值文献筛选
- 实现章节级证据裁剪策略
- 实现记录级 LLM 增强
- 实现规则分、LLM 分与融合评分
- 在分析快照中暴露 LLM 增强摘要

#### `domain`

- 定义 `NormalizedTrialRecord`
- 定义 `NormalizedLiteratureRecord`
- 定义 `AnalysisReadyBundle`
- 定义 `WarningItem`
- 定义记录级 LLM 增强对象与评分对象

#### `repository`

- 建立标准化记录表与分析快照表
- 建立 LLM 增强记录表

#### 测试

- 覆盖字段归一逻辑
- 覆盖重复记录处理
- 覆盖阶段映射与排序逻辑
- 覆盖记录级 LLM 增强、失败回退与融合评分逻辑

### 交付物

- 标准化记录对象与处理逻辑
- 试验/文献统计结果
- 供报告生成使用的 `AnalysisReadyBundle`

### 里程碑 `M2`

系统能够从原始采集结果稳定地产出标准化记录和分析输入，不依赖具体来源字段直接进入报告生成。

### 验收标准

- 双源数据都可转换为统一中间模型
- 去重与关联逻辑可运行并可解释
- 程序能输出阶段分布、主要 sponsor、重点文献列表
- `AnalysisReadyBundle` 能作为独立输入被保存和复用

## 5.4 阶段 3：报告生成与后端 API 闭环

### 阶段目标

打通“分析输入 -> 四章节报告 -> Markdown 输出 -> REST API 返回”的完整后端闭环。

### 范围

- 引入 LLM Provider 抽象
- 实现四个章节的报告生成器
- 实现 Markdown 拼装
- 实现报告查询与来源查询接口

### 具体任务

#### `llm`

- 定义 `BaseLLMProvider`
- 定义 `LLMClient`
- 实现首版 `OpenRouterProvider`
- 封装模型参数、超时与错误处理

#### `report`

- 实现靶点概述生成器
- 实现在研管线概览生成器
- 实现近期研究动态生成器
- 实现竞争格局判断生成器
- 实现 `markdown_renderer.py`

#### `api`

- 实现 `POST /api/reports`
- 实现 `GET /api/reports/{id}`
- 实现 `GET /api/reports/{id}/sources`

#### `repository`

- 建立报告表与来源引用表
- 保存 Markdown 正文、章节摘要、warning 和来源索引

#### `orchestration`

- 实现 `ReportGenerationService`
- 将采集、标准化、分析、报告组装串联起来

#### 测试

- 使用固定 fixture 验证四章节都能生成
- 验证来源引用可以回溯
- 验证部分 source 失败时仍能返回带 warning 的报告

### 交付物

- 完整的报告生成链路
- 对外可调用的 REST API
- 至少一个靶点的可读 Markdown 报告

### 里程碑 `M3`

系统能够通过后端接口生成一份完整报告，并返回结构化结果与来源清单。

### 验收标准

- `POST /api/reports` 可以生成并返回报告
- 报告包含四个必备章节
- 报告中统计数据来自程序计算结果
- 报告 warning 与来源引用可展示

## 5.5 阶段 4：前端展示与 Docker 联调

### 阶段目标

提供面向用户的最小可用 Web 界面，并完成前后端在 Docker 环境中的联调。

### 范围

- 实现报告生成入口页面
- 实现报告详情页面
- 展示 warning 与来源证据
- 完成 Compose 下的前后端联调

### 具体任务

#### 前端页面

- 实现 `TargetSearchForm`
- 实现 `ReportCreatePage`
- 实现 `ReportDetailPage`
- 实现 `ReportViewer`
- 实现 `SourceEvidenceList`

#### 前端能力

- 调用 `POST /api/reports`
- 渲染 Markdown 内容
- 展示加载态、错误态、空态
- 支持复制 Markdown 或下载 `.md`

#### 联调与容器化

- 配置前端访问后端 API 地址
- 确保容器启动后可直接访问前端页面
- 校验 SQLite 和缓存目录挂载生效

### 交付物

- 可直接使用的 Web 页面
- Docker 一键启动联调环境
- 示例查询与展示流程

### 里程碑 `M4`

用户可以通过浏览器输入靶点并查看完整报告，无需手动调用后端接口。

### 验收标准

- 浏览器可访问前端页面
- 页面可以成功提交靶点并展示报告
- 页面能展示来源清单和 warning
- Docker 环境下重启容器后报告数据仍保留

## 5.6 阶段 5：稳定化、测试与交付物完善

### 阶段目标

将系统从“能跑”提升到“能演示、能复现、能验收”的状态。

### 范围

- 完善测试覆盖
- 补齐运行文档与示例报告
- 补强异常处理、日志和配置说明
- 完成验收前自测

### 具体任务

#### 测试

- 完善 Connector 单测
- 完善标准化与分析单测
- 增加报告生成集成测试
- 增加最小端到端冒烟测试

#### 文档

- 补齐运行说明
- 补齐环境变量说明
- 产出至少一个示例报告，如 `HER2`
- 更新架构文档与实施文档中的偏差项

#### 质量补强

- 核查 warning 输出是否清晰
- 核查日志字段是否足够排障
- 核查空结果、超时、单源失败等边界场景

### 交付物

- 可运行源代码
- 架构设计文档
- 实施阶段拆分文档
- 运行说明
- 示例报告
- 基础测试集

### 里程碑 `M5`

V1 原型满足提交与演示条件，可用于汇报、评审和后续迭代规划。

### 验收标准

- 新环境可按文档完成启动
- 至少一个靶点能稳定产出报告
- 核心模块具备基础自动化测试
- 关键异常场景具备清晰输出

## 6. 里程碑之间的依赖关系

各阶段建议按以下依赖推进：


| 里程碑  | 依赖   | 说明                     |
| ---- | ---- | ---------------------- |
| `M0` | 无    | 项目底座必须先完成              |
| `M1` | `M0` | Connector 依赖配置、日志、存储底座 |
| `M2` | `M1` | 标准化与分析依赖真实采集样本         |
| `M3` | `M2` | 报告生成依赖稳定的分析输入          |
| `M4` | `M3` | 前端联调依赖后端 API 基本稳定      |
| `M5` | `M4` | 稳定化与验收基于完整原型进行         |


可并行推进的部分：

- `M1` 后半段，前端可开始搭页面骨架
- `M2` 后半段，可同步准备 Prompt 模板
- `M3` 阶段，前端可提前联调接口契约
- `M4` 阶段，可并行补测试和运行文档

## 7. 建议任务拆解到模块

为便于项目管理，可以将任务进一步映射为 Epic：


| Epic | 对应模块                      | 主要内容                              |
| ---- | ------------------------- | --------------------------------- |
| E1   | `infra + deploy`          | 配置、日志、数据库、Docker、环境初始化            |
| E2   | `connectors + repository` | 双源采集、原始数据入库                       |
| E3   | `domain + normalize`      | 统一模型、标准化、去重、关联                    |
| E4   | `analyze`                 | 统计、排序、裁剪、分析输入构建                   |
| E5   | `llm + report`            | Provider、Prompt、四章节报告、Markdown 输出 |
| E6   | `api`                     | REST 接口、错误处理、响应结构                 |
| E7   | `frontend`                | 表单、报告展示、来源列表、交互状态                 |
| E8   | `tests + docs`            | 自动化测试、运行说明、示例报告、文档完善              |

## 9. 风险前置检查点

实施过程中建议在以下节点设置检查点：

### 检查点 A：阶段 1 结束

重点确认：

- 双源接口是否稳定可用
- 原始记录字段是否足够支撑后续标准化
- 是否需要补充别名解析策略

### 检查点 B：阶段 2 结束

重点确认：

- 标准化字段是否足够支撑四章节报告
- 去重逻辑是否过度合并或遗漏重要记录
- 分析输入体量是否已可控

### 检查点 C：阶段 3 结束

重点确认：

- LLM 输出是否稳定
- 报告是否存在明显幻觉或空泛总结
- warning 和来源引用是否足以支撑可信度

### 检查点 D：阶段 4 结束

重点确认：

- Docker 部署是否稳定
- Web 界面是否满足基本演示需求
- 前后端接口是否还有频繁变更

## 10. 建议的阶段完成定义

一个阶段完成，至少应满足以下条件：

- 代码已合入主线或集成分支
- 相关测试已补齐到当前阶段要求
- 相关文档已更新
- 关键功能已完成一次人工演示
- 已知问题被记录，不影响进入下一阶段

## 11. 结论

对于当前医疗情报系统 V1，最稳妥的落地路径是：

1. 先完成底座与容器化环境
2. 再打通双源采集与原始数据沉淀
3. 然后构建标准化层和分析层
4. 接着完成报告生成与后端接口
5. 最后补前端展示、测试与交付文档

这种拆分方式的优点是：

- 每个阶段都有清晰产物
- 能尽早暴露外部数据源和 LLM 的真实问题
- 能让前后端、数据处理和工程化工作逐步并行
- 能以较低风险交付一个可演示的 V1 原型
